## Manifesto de módulo (Foundry v13)

Fonte: https://foundryvtt.com/article/module-development/ (fetch em 2026-07-03).

- `compatibility`: objeto com `minimum`/`verified`/`maximum` (strings ou números; usamos strings `"13"`).
- `languages[]`: `{ lang, name, path }`.
- `packs[]`: `{ name, label, path, type, ownership? }`. O campo `system` é **obrigatório apenas** para packs de `Actor`, `Item` e `Adventure` (regra desde a v10) — packs de `Macro` não precisam.
- `relationships.requires[]`: `{ id, type, manifest?, compatibility? }`. Omitimos `compatibility` na dependência do Sequencer porque a versão instalada do usuário ainda não foi pinada (aguardando diagnóstico). *(Atualização 2026-07-04: pinada em 4.2.2 — avaliar declarar `compatibility.minimum` na próxima mudança do `module.json`.)*
- `manifest`/`download`: URLs de auto-update; carimbadas pelo CI a partir da tag + `github.repository`, então os placeholders `OWNER` no repo nunca chegam a uma release.

## Sequencer

Fonte: `https://github.com/fantasycalendar/FoundryVTT-Sequencer/releases/latest/download/module.json` (release 4.2.2, fetch em 2026-07-03).

- `id` do módulo: **`sequencer`** (confirmado).
- Manifest estável: `.../releases/latest/download/module.json` (confirmado funcionando).
- Compatibilidade da 4.2.2: minimum 13, verified 14, maximum 14 — cobre nosso alvo v13.

Database API (fonte: `src/modules/sequencer-database.js` no branch master):

- `Sequencer.Database.entries` → objeto keyed por nome de módulo registrado; `Object.keys(entries)` dá os prefixos (`jb2a`, `blfx`, …). Usado na macro de diagnóstico.
- Também disponíveis: `publicModules`, `entryExists(str)`, `getEntry(str)`, `getPathsUnder(path)`, `getAllFileEntries(module)`, `searchFor(path)` — relevantes para o FX Preset Engine (M5) e validação de presets.

## foundryvtt-cli (pipeline de packs)

Fontes: README e `lib/package.mjs` de `github.com/foundryvtt/foundryvtt-cli` (fetch em 2026-07-03). Versão instalada: 3.0.4.

- API Node: `compilePack(src, dest, { log })` — JSON→LevelDB por padrão; `extractPack(src, dest, { log })` para o caminho inverso.
- **`_key` é obrigatório** em cada JSON de source: `lib/package.mjs` faz `if (!doc._key) continue;` — documento sem `_key` é **silenciosamente ignorado**. Formato: `!macros!<_id>`. `_id` deve ter 16 caracteres alfanuméricos.
- Verificado na prática: `npm run packs:build` compilou `bossforgediag001` sem erros nesta máquina (Node 22.18.0).
