## Arquitetura de hooks: state machine do Workflow

O workflow de uso de item/activity é uma máquina de estados (`src/module/Workflow.ts`). Estados (métodos `WorkflowState_*`, enumerados por grep):

`Start`, `AwaitItemCard`, `AoETargetConfirmation`, `AwaitTemplate`, `TemplatePlaced`, `PreambleComplete`, `WaitForAttackRoll`, `AttackRollComplete`, `ConfirmRoll`, `RollConfirmed`, `ValidateRoll`, `WaitForDamageRoll`, `DamageRollStarted`, `DamageRollComplete`, `DamageRollCompleteCancelled`, `WaitForSaves`, `SavesComplete`, `WaitForUtilityRoll`, `UtilityRollComplete`, `AllRollsComplete`, `ApplyDynamicEffects`, `RollFinished`, `Completed`, `Cleanup`, `Cancel`, `Abort`, `Suspend`, `NoAction`, `Default`.

**Para cada transição de estado** o Midi dispara hooks **canceláveis** com o workflow como payload (`callHooksForAction`, `Workflow.ts` ~1957–1964, chamado em ~2221 e ~2235):

- `midi-qol.pre<Estado>` — antes de entrar no estado (retornar `false` aborta para `RollFinished`/`Abort`);
- `midi-qol.post<Estado>` — depois do estado.

Os "macro passes" de OnUse usam os mesmos nomes (`allHooks = onUseMacroOptions + stateHooks`).

## O que interessa ao M2 (resistência lendária)

- Estados de save: **`WaitForSaves` → `SavesComplete`** ⇒ hooks `midi-qol.preSavesComplete` / `midi-qol.postSavesComplete`.
- Dados no workflow (`Workflow.ts` ~1275–1334): `workflow.saves: Set<Token>` e `workflow.failedSaves: Set<Token>` (inicializado como `new Set(this.targets)` e recalculado ao avaliar saves). Aplicação de dano posterior usa esses conjuntos.
- Hook por alvo: `midi-qol.preTargetSave` (via `asyncHooksCallAll`) — candidato para interceptar o save de UM alvo específico (o boss) antes da avaliação.
- Também existem: `midi-qol.preTargetDamageApplication`, `midi-qol.ReactionFilter`, `midi-qol.setReactionUsed`/`setBonusActionUsed` (+ variantes `pre`).
- Hooks nomeados "clássicos" continuam em uso interno (o próprio Midi escuta): `midi-qol.AttackRollComplete`, `midi-qol.DamageRollComplete` (`midi-sounds.ts:162,180`).
- Ciclo de vida: `midi-qol.midiReady`, `midi-qol.ready`, `midi-qol.setup`, `midi-qol.configSettingsChanged`.

## Estratégia candidata para o M2 (validar na implementação)

1. Hook em `midi-qol.preSavesComplete` (ou `preTargetSave`) **no cliente do GM**: se um boss com `legres.value > 0` está em `failedSaves`, abrir o dialog "queimar resistência lendária?".
2. Ao confirmar: mover o token do boss de `workflow.failedSaves` para `workflow.saves` e decrementar `system.resources.legres.spent` (avaliar reutilizar o `resistSave()` nativo do dnd5e — depende de como o Midi monta os flags `dnd5e.roll` nos cards dele; pode ser necessário update direto + ajuste do card do Midi).
3. O hook é cancelável e assíncrono (`asyncHooksCall`) — dá para aguardar o dialog dentro dele. Confirmar na versão instalada que o workflow aguarda o retorno da Promise.

## Riscos/aberto

- Nome/assinatura podem divergir na versão instalada do usuário (regra de ouro). *(Atualização 2026-07-04: versão pinada em 13.0.63 — falta só o cruzamento branch-head → release ao abrir o M2.)*
- Verificar se `preSavesComplete` roda no cliente que executa o workflow (normalmente o do usuário que usou o item — pode ser um PLAYER no caso de magia de PC contra o boss). O M2 talvez precise de interceptação GM-side coordenada (o Midi tem `socketlibSocket`/GMAction para ações GM-side; nada player-facing no nosso MVP → o dialog deve aparecer para o GM; estudar se o hook roda no cliente do jogador e como sinalizar o GM sem socketlib próprio… ou aceitar socketlib como dependência já presente via Midi).
